# Объяснение решения задачи прогнозирования отказа оборудования
Реализован полный ML-пайплайн в одном скрипте `ml_solution.py`. Основные этапы:

1. **Загрузка данных** - из UCI по прямой ссылке.
2. **Анализ данных** - размер, типы, пропуски, распределение целевой переменной.
3. **Предобработка**:
   - Удаление идентификатора `UDI`.
   - Извлечение типа продукта из `Product ID` (буква L, M, H) или из колонки `Type`.
   - Кодирование категориального признака `ProductType` через `LabelEncoder`.
   - Удаление столбцов `TWF, HDF, PWF, OSF, RNF` - это индикаторы конкретных типов отказов. Если их оставить, модель будет «читерить», предсказывая отказ прямо по этим колонкам. В реальности таких меток нет, поэтому для обобщающей способности они удаляются. Используется только итоговая метка `Machine failure`.
   - Масштабирование числовых признаков (`StandardScaler`) для корректной работы XGBoost и Random Forest (хотя деревьям масштабирование не критично, но для единообразия оставлено).
4. **Разделение данных** - стратифицированное 80/20 (train/validation) для сохранения доли отказов в обеих выборках.
5. **Обучение моделей**:
   - **Random Forest** (лес решающих деревьев) - устойчив к переобучению, хорошо работает с табулярными данными.
   - **XGBoost** (градиентный бустинг) - более мощный, часто даёт лучшую точность, но требует настройки дисбаланса классов.
   - Для борьбы с сильным дисбалансом (≈3.4% отказов) использованы: `class_weight='balanced'` в Random Forest и `scale_pos_weight` в XGBoost.
6. **Оценка качества** на валидационной выборке по метрикам: Accuracy, Precision, Recall, F1, ROC-AUC.
7. **Генерация алертов** - для каждого образца вычисляется вероятность отказа, и если она ≥ 0.05, выставляется `alert = True`. Выводится таблица топ-20 образцов с наибольшей вероятностью.

## Выбор основной метрики - Recall (полнота)
В задаче предиктивного обслуживания **цена пропуска отказа** (False Negative) намного выше, чем цена ложной тревоги (False Positive). Пропущенный отказ приводит к аварийной остановке, дорогому ремонту и потерям производства. Ложная тревога всего лишь вызывает внеплановую проверку.

Поэтому **основная метрика - Recall**: доля реальных отказов, которые модель успешно обнаружила.

Дополнительно контролируется Precision, чтобы ложные срабатывания не были слишком частыми. ROC-AUC даёт общее представление о разделяющей способности.

## Результаты
| Модель         | Accuracy | Precision | Recall | F1     | ROC-AUC |
|----------------|----------|-----------|--------|--------|---------|
| Random Forest  | 0.9830   | 0.7931    | 0.6765 | 0.7302 | 0.9657  |
| XGBoost        | 0.9860   | 0.8030    | 0.7794 | 0.7910 | 0.9688  |

**Интерпретация:**  
- XGBoost показывает более высокий Recall (0.78 против 0.68), что соответствует цели задачи.  
- Precision около 0.80 означает, что из 10 предупреждений 8 будут реальными отказами (ошибка в 2).  
- Accuracy высокая из-за сильного дисбаланса, но она не является показательной.

## Генерация алертов (пример из вывода)
- Всего алертов (вероятность ≥ 0.05): **121** из 2000 валидационных образцов.
- Правильных алертов (реальный отказ): **59** (Recall = 59/68 = 0.867? - но Recall модели был 0.78, значит часть отказов не попала в топ-20, но в общем списке их 59 из 68? Проверим: 68 отказов в валидации, модель нашла 59, Recall = 0.867, это даже выше, чем 0.779 - возможно, потому что порог 0.05 даёт больше True Positive, чем стандартное предсказание класса. Отлично, порог 0.05 увеличивает Recall ценой большего числа False Positive - 62 ложных алерта.)
- **Вывод:** Порог 0.05 оправдан - позволяет поймать 87% отказов, хотя и даёт 62 ложные тревоги.
